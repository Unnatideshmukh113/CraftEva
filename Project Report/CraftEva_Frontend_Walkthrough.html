Line-by-Line

L1StrictMode — React dev helper that highlights potential problems (double-renders in dev, warns about deprecated APIs)
L2createRoot — React 18+ concurrent mode API (replaces ReactDOM.render). Makes React capable of interruptible rendering.
L4getElementById('root') — finds the <div id="root"> in index.html — this is the single DOM entry point for the entire SPA
L5.render(<App/>) — mounts the entire React component tree starting from App, replacing the #root div's contents
What is StrictMode?
StrictMode is a development tool that deliberately double-invokes certain functions (like render and useEffect cleanup) to expose side-effects. It doesn't affect production builds. It helps you catch bugs like missing cleanup in useEffects, deprecated lifecycle methods, and unexpected re-renders.
What is createRoot vs ReactDOM.render?
createRoot is React 18's concurrent mode API. It enables features like automatic batching, transitions, and Suspense. The old ReactDOM.render was synchronous and blocking — createRoot allows React to interrupt and resume rendering work, making the UI more responsive.
What is a Single Page Application (SPA)?
In an SPA, the server sends one HTML file (index.html with <div id="root">) and React takes over all rendering in the browser. Navigation between pages doesn't reload the page — React Router intercepts URL changes and swaps components. This makes the app feel fast like a native app.

Why this provider nesting order?

AuthProvider is outermost because CartProvider and WishlistProvider both call useAuth() — they must be inside AuthProvider
CartProvider is inside WishlistProvider because both are used independently — no dependency between them
Router wraps everything because Header, Footer, and all pages need useNavigate/useLocation
Why do we nest Context Providers?
Each Context.Provider only makes its value available to its children. If CartProvider uses useAuth() internally, CartProvider must be a child of AuthProvider — otherwise useContext(AuthContext) would return undefined. The nesting order mirrors the dependency graph.
What is BrowserRouter / Router?
BrowserRouter uses the HTML5 History API (pushState) to keep the UI in sync with the URL without page reloads. It provides routing context (location, history) to all child components. useNavigate() and useParams() only work inside a Router — that's why Router wraps the entire app.
What does Routes and Route do?
Routes renders only the first Route whose path matches the current URL. Route maps a URL path to a component. Without Routes, all matching Routes would render simultaneously. The replace attribute (used in Navigate) prevents the redirect from creating a history entry — so Back button doesn't loop.
Where is index.html and what does it contain?
In a Vite project, index.html is at the project root. It contains <div id="root"></div> (the mount point) and a <script type="module" src="/src/main.jsx"> tag. Vite serves this file for any URL request — React Router then renders the correct component based on the URL path.

Why sessionStorage instead of localStorage?
sessionStorage clears automatically when the browser tab is closed — so the user is logged out when they close the tab, which is more secure. localStorage persists until explicitly cleared, meaning the user stays logged in even after closing the browser — good for "remember me" features but a security risk on shared computers. CraftEva chooses the more secure sessionStorage approach.
Why the loading state? What happens without it?
On page refresh, the useEffect that restores the session runs asynchronously. Without loading state, ProtectedRoute checks isAuthenticated before the session is restored — finding null user — and redirects to /login even for authenticated users. The loading flag makes ProtectedRoute show "Loading..." until the session check completes, then render correctly.
What is a custom hook? Why useAuth()?
A custom hook is a function starting with "use" that calls other hooks. useAuth() wraps useContext(AuthContext) and adds the null guard. Instead of importing and calling useContext(AuthContext) in every component, you just call useAuth(). The null guard gives a helpful error if someone forgets to wrap their component in AuthProvider, instead of a cryptic "cannot read property of undefined" error.
What does !!user mean?
!! is double negation — it converts any value to a boolean. If user is null, !null is true, !!null is false. If user is an object, !{...} is false, !!{...} is true. So !!user is true when user exists and false when it's null — a clean way to derive isAuthenticated without an explicit boolean state.
Why store role in both the user object AND a separate sessionStorage key?
The user object in sessionStorage is a JSON string that may need parsing. Storing role separately allows the Axios interceptor and ProtectedRoute to access the role as a plain string without parsing JSON — a small performance and convenience optimization. Both are kept in sync on login and cleared on logout.
What is optional chaining (?.) used in user?.role?
Optional chaining safely accesses a property of an object that might be null/undefined. user?.role returns undefined (not an error) when user is null. Without it, user.role when user is null would throw "TypeError: Cannot read properties of null". This is essential here since user is null before login.

api/axios.js — HTTP Client with Interceptors SecurityAPI

import axios from 'axios';

// Create a pre-configured Axios instance
const api = axios.create({
  baseURL: 'http://localhost:8080/api', // Prepended to every request URL
  headers: { 'Content-Type': 'application/json' },
});

// ── REQUEST INTERCEPTOR ──────────────────────────────────────────
// Runs BEFORE every request is sent
api.interceptors.request.use(
  (config) => {
    const token = sessionStorage.getItem('token'); // Read fresh token every time
    if (token) {
      config.headers.Authorization = `Bearer ${token}`; // Attach to header
    }
    return config;  // ← MUST return config or request is cancelled
  },
  (error) => Promise.reject(error) // Propagate request setup errors
);

// ── RESPONSE INTERCEPTOR ─────────────────────────────────────────
// Runs AFTER every response is received
api.interceptors.response.use(
  (response) => response,  // ← 2xx responses: pass through unchanged
  (error) => {
    if (error.response && error.response.status === 401) {
      // Token expired or missing → clear session → force re-login
      sessionStorage.removeItem('token');
      sessionStorage.removeItem('user');
      sessionStorage.removeItem('role');
      window.location.href = '/login'; // Hard redirect (not useNavigate — no Router context)
    }
    return Promise.reject(error); // Re-throw for per-call .catch() handlers
  }
);

export default api;
Why create an Axios instance instead of using axios directly?
An Axios instance (axios.create()) lets you set a baseURL and default headers once. Without it, every API call would need the full URL (http://localhost:8080/api/auth/login) and manually add headers. The instance centralises configuration — change the base URL in one place when deploying to production.
How do Axios interceptors work?
Interceptors are middleware for HTTP requests/responses. The request interceptor runs before the request is sent — great for attaching auth tokens. The response interceptor runs after a response is received — great for handling global errors like 401. Every API call made through the api instance goes through both interceptors automatically.
Why use window.location.href instead of useNavigate for the 401 redirect?
useNavigate() is a React Router hook and requires the component to be inside a Router context. The Axios interceptor is plain JavaScript — it runs outside any React component context, so hooks cannot be used. window.location.href = '/login' is the only way to redirect from non-component code. It also triggers a full page reload which clears all React state — a clean logout.
Why read the token from sessionStorage on every request instead of once?
If we closed over the token at module load time (const token = sessionStorage.getItem('token') outside the interceptor), we'd capture null (user not logged in yet). Reading it inside the interceptor function ensures we always get the current token — even after login happens later. This is the correct approach for dynamic auth state.
What does return Promise.reject(error) do?
It re-throws the error after the interceptor handles it. If we just returned nothing, Axios would treat the response as success. By rejecting the promise, the error propagates to the .catch() block of the individual API call — so each page can still show its own "Network error" message in addition to the global 401 handling.

ProtectedRoute.jsx — Route Guard RoutingSecurity

import { Navigate } from 'react-router-dom';
import { useAuth } from '../context/AuthContext';

const ProtectedRoute = ({ children, requiredRole = null }) => {
  const { isAuthenticated, user, loading } = useAuth();

  // Step 1: Wait for session restore (prevents flash-redirect on refresh)
  if (loading) return <div>Loading...</div>;

  // Step 2: Not logged in → redirect to login
  if (!isAuthenticated) return <Navigate to="/login" replace />;

  // Step 3: Logged in but wrong role → redirect to home
  if (requiredRole && user?.role !== requiredRole) return <Navigate to="/" replace />;

  // Step 4: Authenticated + correct role → render the page
  return children;
};

Decision Tree

loading=true → show spinner (prevents wrong redirect during hydration)
isAuthenticated=false → <Navigate to="/login" replace />
requiredRole="SELLER" but user.role="BUYER" → <Navigate to="/" replace />
All checks pass → render {children} (the actual page component)
What does the replace prop on Navigate do?
replace={true} replaces the current entry in the browser history stack instead of pushing a new one. Without replace, clicking Back would take the user back to the protected route, which would redirect again — creating an infinite loop. With replace, going back skips the protected URL entirely.
What is the flash-of-wrong-content problem and how is loading solving it?
On page refresh, React renders before useEffect runs. The user object is null for a brief moment while sessionStorage is being read. Without loading state, ProtectedRoute sees isAuthenticated=false and redirects to /login — even for a logged-in user. The loading flag delays that check until hydration from sessionStorage is complete.
What is children prop here?
In React, children is a special prop that represents whatever JSX is placed between the opening and closing tags of a component. Here, <ProtectedRoute requiredRole="BUYER"><Cart/></ProtectedRoute> — "Cart" is the children prop. ProtectedRoute acts as a wrapper that conditionally renders its children.
Can a SELLER access /cart if they manually type the URL?
No — ProtectedRoute checks user?.role !== requiredRole ("SELLER" !== "BUYER" → true) and redirects to "/". Client-side route guards prevent UI access, but the backend is the real security layer — the API calls from /cart page use Bearer tokens, and the backend's @PreAuthorize("hasRole('BUYER')") would reject SELLER tokens with 403.

CartContext.jsx — Cart State & API Integration Context

// CartContext uses 3 useEffect hooks:

// useEffect 1: Cleanup corrupted cartOrderId on mount
useEffect(() => {
  const storedOrderId = sessionStorage.getItem('cartOrderId');
  if (storedOrderId && (storedOrderId.includes(':') || isNaN(parseInt(storedOrderId)))) {
    sessionStorage.removeItem('cartOrderId'); // Clear invalid data like "10:1"
    setCartOrderId(null); setCartItems([]);
  }
}, []);  // once on mount

// useEffect 2: Load cart from backend when user logs in
useEffect(() => {
  if (user && isBuyer) {
    api.get(`/order-items/order/${orderId}`)   // Fetch existing cart items
       .then(res => setCartItems(res.data));
  } else {
    setCartItems([]); setCartOrderId(null);    // Non-buyers get empty cart
  }
}, [user, isBuyer]);  // re-runs when user changes (login/logout)

// useEffect 3: Sync cart state to sessionStorage
useEffect(() => {
  if (cartItems.length > 0) sessionStorage.setItem('cartItems', JSON.stringify(cartItems));
  else sessionStorage.removeItem('cartItems');
  if (cartOrderId) sessionStorage.setItem('cartOrderId', cartOrderId.toString());
  else sessionStorage.removeItem('cartOrderId');
}, [cartItems, cartOrderId]);  // runs whenever cart changes

// addToCart flow:
const addToCart = async (productId, quantity = 1) => {
  // 1. Validate user is logged in as BUYER
  if (!user || !isBuyer) return { success: false, error: '...' };

  // 2. Create order if none exists yet
  if (!orderId) {
    const res = await api.post('/orders/add', { buyer: { userId: user.userId }, totalAmount: 0 });
    orderId = res.data.orderId;
    setCartOrderId(orderId.toString());
  }

  // 3. Add item to order via /order-items/add
  await api.post('/order-items/add', { product: { productId }, order: { orderId }, quantity, price: 0 });

  // 4. Refresh cart items from backend
  const updated = await api.get(`/order-items/order/${orderId}`);
  setCartItems(updated.data);
};
Why does addToCart create an order before adding an item?
The backend design links OrderItems to an Order (order_id FK). There's no "shopping cart" table — the cart IS an order with status PLACED. Before any item can be added, an order row must exist. So the first "add to cart" action triggers POST /orders/add (creating an Order with totalAmount=0), then POST /order-items/add links the product to that order.
Why are there 3 useEffect hooks instead of one?
Each useEffect has a different purpose and dependency: 1) Cleanup runs once on mount (no deps). 2) Cart loading runs when the user changes (login/logout triggers it). 3) sessionStorage sync runs whenever cartItems or cartOrderId change. Combining them would create circular dependencies and incorrect timing. Separating them makes each effect's intent clear and correct.
Why does useEffect 2 have [user, isBuyer] as dependencies?
When a user logs in, the user object changes from null to {userId, role, ...} — this triggers useEffect 2 to load their cart. When they log out, user becomes null again — the else branch clears the cart. Without user in the dependency array, the cart would never reload after login (stale closure problem).
What is the stale closure problem in useEffect?
If you use a variable inside useEffect but don't include it in the dependency array, useEffect captures the initial value and never sees updates — this is a stale closure. For example, if user is not in deps, the cart loading code would always use the initial null value even after login. ESLint's exhaustive-deps rule helps catch this.
What does getCartTotal() do and why not store it as state?
getCartTotal() calculates the sum from cartItems dynamically: cartItems.reduce((total, item) => total + item.price, 0). Storing total as separate state would require keeping it in sync manually — a common source of bugs. Since total is derived entirely from cartItems, computing it on-demand is simpler and always correct. This is the "derive, don't duplicate" principle in React state management.

🎤 All Possible Frontend Interview Questions

React Core

What is JSX and how does it work?
JSX is a syntax extension that allows HTML-like markup in JavaScript. React's build tool (Babel/Vite) compiles JSX to React.createElement() calls. For example, <div className="x"> becomes React.createElement('div', {className: 'x'}). Attributes use camelCase (className not class, onClick not onclick) because JSX compiles to JS object keys.
What is the Virtual DOM?
The Virtual DOM is React's in-memory JavaScript copy of the real DOM. When state changes, React creates a new Virtual DOM, diffs it against the previous one (reconciliation), computes the minimal changes, and applies only those to the real DOM in a single batch. This is far faster than directly manipulating the DOM on every state change.
What are React hooks and what rules apply to them?
Hooks are functions (starting with "use") that let functional components use React features like state and lifecycle. Rules: 1) Only call at the top level — never inside if/loops/nested functions. 2) Only call in React functions or custom hooks — not regular JS. These rules ensure React can maintain the order of hook calls across renders (React tracks hooks by call order).
Explain useState.
useState(initialValue) returns [value, setter]. Calling the setter schedules a re-render with the new value. The setter can take a value directly or a function: setCount(prev => prev + 1) — the function form is safe when next state depends on previous, avoiding stale closure bugs in async code.
Explain useEffect and its dependency array.
useEffect(fn, deps) runs the function after render. Empty array [] = run once on mount. With deps = run when any dep changes. No array = run after every render (rarely useful). The returned function is the cleanup — runs before the next effect and on unmount. Used for API calls, event listeners, timers.
What is prop drilling and how does Context API solve it?
Prop drilling is passing props through many intermediate components that don't need them, just to reach a deep child. Context API creates a shared store at a high level (AuthContext, CartContext) and any descendant can read it via useContext(). CraftEva uses this so Cart badge in Header, product pages, and payment page all read from the same CartContext without prop chains.
What is useMemo and useCallback?
useMemo(fn, deps) caches the return value of fn until deps change — prevents expensive recalculation on every render. useCallback(fn, deps) caches the function reference itself — prevents child components wrapped in React.memo from re-rendering when the parent re-renders (because the function reference stays the same). Use when profiling shows performance issues, not by default.
What is React.memo?
React.memo is a higher-order component that wraps a component and skips re-rendering if its props haven't changed (shallow comparison). Useful for expensive components that receive stable props. Example: ProductCard wrapped in React.memo won't re-render when the parent re-renders if the product prop hasn't changed.

React Router v6

How does React Router work?
React Router uses BrowserRouter (History API) to listen to URL changes. When the URL changes, Routes compares the path to each Route's path and renders the matching component. Navigation with <Link> or useNavigate() updates the URL without a page reload. The actual HTML file never changes — React swaps components in the div#root.
What is the difference between Link and useNavigate?
Link renders as an <a> tag and is used in JSX for user-clickable navigation. useNavigate() returns a navigate function used programmatically — e.g., after a form submit, API success, or logout. Both update the URL via History API without a page reload.

State Management

When would you use Redux instead of Context API?
Context API is good for low-frequency updates (auth state, theme). For high-frequency updates like rapid state changes (typing in a search field updates 100 results), Context causes every consumer to re-render. Redux with selectors re-renders only components whose selected slice changed. Also: Redux DevTools make debugging time-travel easier in complex apps. For CraftEva's scale, Context is the right choice.
What is the difference between controlled and uncontrolled components?
Controlled: form input value is managed by React state — every keystroke calls setState, React controls the displayed value. Uncontrolled: value is in the DOM, read via useRef when needed. CraftEva's Login form is controlled — const [email, setEmail] = useState('') with onChange={e => setEmail(e.target.value)}. Controlled components are preferred because they enable live validation and conditional submit buttons.

Security & Auth

Is storing JWT in sessionStorage safe?
sessionStorage is accessible by JavaScript running on the page, making it vulnerable to XSS (Cross-Site Scripting) attacks — if an attacker injects JS, they can steal the token. The safest option is httpOnly cookies (not accessible by JS). However, sessionStorage is safer than localStorage (clears on tab close) and is acceptable for many applications. CraftEva mitigates risk by ensuring no user-generated content is rendered as raw HTML.
What is XSS and how do you prevent it?
XSS (Cross-Site Scripting) is an attack where malicious JavaScript is injected into a webpage. React prevents XSS by default — it escapes all values rendered inside JSX using dangerouslySetInnerHTML is avoided. The risk exists when rendering raw HTML from user input. Prevention: never use dangerouslySetInnerHTML, sanitize inputs, use Content Security Policy headers.

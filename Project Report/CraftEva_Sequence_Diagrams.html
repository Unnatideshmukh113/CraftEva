🔑 Why JWT over sessions?

JWT is stateless — the server stores no session data. Any server instance can validate a token using just the secret key, making this horizontally scalable. Sessions require shared state (e.g. Redis) across servers.

🔒 Why BCrypt?

BCrypt is a one-way adaptive hash — it includes a salt and a cost factor. Unlike SHA-256, it's intentionally slow and resists brute-force. Even if the DB is leaked, passwords can't be reversed.

📦 Why store role in JWT claims?

The JwtAuthenticationFilter reads the "role" custom claim to set the ROLE_BUYER authority. This way no DB lookup is needed on every request — everything needed for authorization is inside the token.

⚙️ permitAll on /api/auth/**

SecurityConfig marks the entire /api/auth/** path as public — JwtAuthenticationFilter still runs but passes through if no token, so login works without requiring a token.
🎤 SAY IN INTERVIEW:
"On login, the user sends email and password. My UserServiceImpl fetches the user, compares the password using BCrypt's matches() method, and if valid, calls JwtUtil.generateToken() which creates an HMAC-SHA256 signed token with the user's email as the subject and their role as a custom claim. This token is returned to the frontend and stored in localStorage. Every subsequent request attaches this token in the Authorization header, and my JwtAuthenticationFilter validates it before any controller is reached."

③ Place Order Flow (Most Important)

Browser
(Cart.jsx / Payment.jsx)
JwtFilter
OrderController
OrderServiceImpl
@Transactional
UserRepository
OrderRepository
MySQL DB
  • 1
    Browser → JwtFilter
    POST /api/orders/addPOST
    Header: Authorization: Bearer eyJ0eXAiOiJKV1QiLCJhbGci...
    Body: {"buyer":{"userId":42},"totalAmount":850.00,"orderItems":[...]}
  • 2
    JwtAuthenticationFilter
    extractUsername(jwt) → "buyer@email.com"
    validateToken(jwt, userDetails) → true
    getRoleFromToken(jwt) → "BUYER"
    Set SecurityContextHolder with authority ROLE_BUYER
  • 3
    OrderController
    @PreAuthorize("hasRole('BUYER') or hasRole('ADMIN')")
    @Valid validates OrderDto before entering service
    service.placeOrder(orderDto)
    • @Transactional BEGIN
  • 4
    OrderServiceImpl → UserRepository → MySQL
    userRepository.findById(42L) → Optional<User> buyer
    SELECT * FROM users WHERE user_id = 42
    Throws RuntimeException("Buyer not found") if not present
  • 5
    OrderServiceImpl — Build Order entity
    order.setOrderDate(LocalDate.now())
    order.setStatus(OrderStatus.PLACED)
    order.setTotalAmount(850.00)
    order.setBuyer(buyer)
  • 6
    OrderServiceImpl → OrderRepository → MySQL
    orderRepository.save(order)
    INSERT INTO orders (order_date, status, total_amount, buyer_id) VALUES ('2026-02-27', 'PLACED', 850.00, 42)
    Returns saved Order with auto-generated orderId = 101
  • 7
    → Browser
    200 OK — OrderDtoRETURN
    Body: {"orderId":101,"orderDate":"2026-02-27","status":"PLACED","totalAmount":850.00,"buyer":{...}}
    Frontend navigates to Orders page — user can now download invoice from GET /api/invoices/order/101
ALT — Error Paths
  • E1
    No token / expired token → 401 (JwtFilter does not set SecurityContext → Spring Security blocks)
  • E2
    Token has role=SELLER → 403 Forbidden (@PreAuthorize("hasRole('BUYER')...") fails)
  • E3
    buyerId not found in DB → RuntimeException("Buyer not found") → @Transactional rolls back

🔄 Why @Transactional?

The entire placeOrder() method is wrapped in a DB transaction. If anything fails after the order is INSERTed, JPA rolls back ALL changes — preventing orphaned incomplete order records.

📋 DTO Pattern

We never expose the raw Order entity to the frontend. OrderDto controls exactly what fields are sent — preventing accidental exposure of sensitive fields and circular JSON references.

🔐 @PreAuthorize

@PreAuthorize("hasRole('BUYER') or hasRole('ADMIN')") checks the authority in SecurityContextHolder — set by JwtAuthenticationFilter using the role claim from the JWT.

📅 LocalDate vs LocalDateTime

Order uses LocalDate for order_date (day precision). Payment uses LocalDateTime for payment_date (exact timestamp). Both are Hibernate-compatible without extra config in Spring Boot 3.
🎤 SAY IN INTERVIEW:
"When a buyer places an order, the request first goes through JwtAuthenticationFilter which validates the token and sets the user's role. Then @PreAuthorize checks the role. In OrderServiceImpl, the placeOrder() method is @Transactional — it fetches the buyer, creates the Order entity with status PLACED, and saves it. If any step fails, the transaction automatically rolls back, ensuring no incomplete order records in the database."

⑥ Error Flows — 400 / 401 / 403 / 404 / 500

401 Unauthorized

  • 1
    Request arrives with no Authorization header OR expired JWT
  • 2
    JwtAuthenticationFilter: authHeader == null → filterChain.doFilter() without setting SecurityContext
  • 3
    Spring Security: protected endpoint + no auth → 401 Unauthorized
  • 4
    Frontend Axios: catches 401 → clears localStorage → navigate('/login')

403 Forbidden

  • 1
    Request: Valid JWT but user has role=BUYER trying to reach POST /api/products
  • 2
    JwtFilter: sets SecurityContext with ROLE_BUYER
  • 3
    @PreAuthorize("hasRole('SELLER') or hasRole('ADMIN')") → evaluates false
  • 4
    AccessDeniedException → 403 Forbidden returned to client

404 Not Found

  • 1
    GET /api/invoices/order/9999 — orderId 9999 does not exist
  • 2
    orderRepository.findById(9999L) returns Optional.empty()
  • 3
    .orElseThrow(() → RuntimeException("Order not found"))
  • 4
    Unhandled → 500. Improvement: use custom NotFoundException → 404

400 Bad Request / 409 Conflict

  • 1
    400: POST /api/auth/register with blank email field → @Valid → MethodArgumentNotValidException → 400
  • 2
    409: POST /api/auth/register with email that already exists → RuntimeException("Email already registered") → currently 500 (should be 409)
  • 3
    Improvement: Add @ControllerAdvice GlobalExceptionHandler to map each exception type to correct HTTP status
🎤 SAY IN INTERVIEW:
"401 means the token is missing or expired — no authentication at all. 403 means the token is valid but the role is wrong — the user is authenticated but not authorized. In my app, 401 happens when JwtAuthenticationFilter doesn't set the SecurityContext, and 403 happens when @PreAuthorize fails. Currently my RuntimeExceptions map to 500 by default — an improvement would be a @ControllerAdvice GlobalExceptionHandler to map ResourceNotFoundException to 404 and duplicate email to 409."
